# WhereIs System Requirements

## 1. Overview

WhereIs เป็นระบบบริหารจัดเก็บ ค้นหา ติดตาม และจัดการวงจรชีวิตของสิ่งของภายใน Workspace เช่น บ้าน สำนักงาน บริษัท ร้านค้า โกดัง หรือองค์กร

ระบบต้องตอบคำถามหลักได้ว่า:

* มีของอะไร
* อยู่ที่ไหน
* มีเท่าไร
* อยู่กับใคร
* ใครกำลังใช้งาน
* ใครรอใช้งานต่อ
* ต้องคืนเมื่อไร
* ใกล้หมดอายุหรือถึงกำหนดอะไร
* เคยถูกยืม เบิก ย้าย ซ่อม แจ้งหาย หรือจำหน่ายเมื่อไร
* ใครเป็นผู้ดำเนินการในแต่ละขั้นตอน

## 3. Workspace

Workspace คือพื้นที่ข้อมูลหลักของแต่ละกลุ่มผู้ใช้งาน เช่น:

* บ้านของฉัน
* บริษัท ABC
* โกดังอะไหล่
* ร้านอาหาร
* สำนักงานสาขาเชียงใหม่

ผู้ใช้หนึ่งคนสามารถอยู่ได้หลาย Workspace และแต่ละ Workspace ต้องแยกข้อมูลออกจากกัน

### Requirements

* ผู้ใช้สามารถสร้าง Workspace ได้
* ผู้ใช้สามารถแก้ไข Workspace ได้ตามสิทธิ์
* ผู้ใช้สามารถลบ Workspace ได้ตามสิทธิ์
* ผู้ใช้สามารถสลับ Workspace ได้
* Workspace มีสมาชิกของตัวเอง
* Workspace มี Role และ Permission ของตัวเอง
* Workspace มี Container และ Item ของตัวเอง
* ข้อมูลแต่ละ Workspace ห้ามรั่วข้ามกัน

## 5. Role และ Permission

ระบบใช้แนวคิด:

```text
Primary Role
+
Extra Permissions
+
Container Access Scope
```

Role เป็นชุดสิทธิ์พื้นฐาน ส่วน Extra Permissions ใช้เพิ่มหรือลดสิทธิ์เฉพาะบุคคล

### Core Rule

ไม่มี Self-Service Flow แยก

ทุกองค์กรใช้ Workflow เดียวกัน แต่คนที่มีสิทธิ์หลายอย่างสามารถทำหลายขั้นตอนเองได้

ตัวอย่างบริษัทใหญ่:

```text
A ขอเบิก
B อนุมัติ
C หยิบของ
C ส่งมอบ
A รับของ
```

ตัวอย่างบริษัทเล็ก:

```text
A ขอเบิก
A อนุมัติ
A หยิบของ
A ส่งมอบ
A รับของ
```

ถ้า A มี Permission ครบ สามารถทำทุกขั้นตอนได้

### Self Approval

ผู้ใช้สามารถอนุมัติรายการที่ตนเองสร้างได้ ถ้ามี Permission สำหรับ Action นั้น

## 6. Container

Container คือพื้นที่หรือตำแหน่งจัดเก็บ ไม่ได้หมายถึงแค่กล่อง

ตัวอย่าง Container:

* สาขา
* อาคาร
* ชั้น
* ห้อง
* โซน
* โกดัง
* ลานจอดรถ
* ตู้
* ชั้นวาง
* กล่อง
* ลิ้นชัก
* ตู้เย็น
* ช่องแช่แข็ง

Container สามารถซ้อนกันได้ไม่จำกัดระดับ

ตัวอย่าง:

```text
บริษัท ABC
└── สำนักงานใหญ่
    └── อาคาร A
        └── ชั้น 2
            └── ห้อง IT
                └── ตู้ A
                    └── ชั้นวาง 2
                        └── กล่องแดง
```

### Requirements

* สร้าง Container ได้หลายระดับ
* ย้าย Container ได้ตามสิทธิ์
* แก้ไข Container ได้ตามสิทธิ์
* ลบ Container ได้ตามสิทธิ์
* กำหนดว่าใครเห็น Container ใดได้
* กำหนดว่าใครจัดการ Container ใดได้
* การค้นหาต้องแสดงเฉพาะของใน Container ที่ผู้ใช้มีสิทธิ์เห็น

---

## 7. Container Access Control

Workspace Permission ตอบว่า:

```text
ทำอะไรได้
```

Container Access ตอบว่า:

```text
ทำได้กับพื้นที่ไหน
```

ตัวอย่าง:

User A มี Permission:

* ดูของ
* ขอยืม
* คืนของ
* แจ้งหาย

แต่มี Container Access เฉพาะ:

```text
โกดัง A > โซนเครื่องมือ
```

ดังนั้น A จะค้นหา ดู ยืม คืน หรือแจ้งหายได้เฉพาะของใน Container นั้น

* กำหนดสำคัญ

### Quantity Item Examples

## 9. Unit Management

ระบบต้องรองรับหลายหน่วย และการแปลงหน่วยต้องผูกกับ Item นั้น ๆ

ตัวอย่าง:

```text
ตะปู 2 นิ้ว
Base Unit: ตัว
1 กล่อง = 500 ตัว
```

```text
ถ่าน AA
Base Unit: ก้อน
1 กล่อง = 24 ก้อน
```

```text
ไข่ไก่
Base Unit: ฟอง
1 แผง = 30 ฟอง
```

```text
กระดาษ A4
Base Unit: แผ่น
1 รีม = 500 แผ่น
1 ลัง = 5 รีม
```

### Rule

ห้ามกำหนด Conversion กลางว่า 1 กล่อง = จำนวนเท่ากันทุก Item

เพราะคำว่า “กล่อง” ของแต่ละ Item มีจำนวนไม่เท่ากัน

ระบบต้องเก็บยอดจริงเป็น Base Unit เสมอ

## 10. Batch / Lot / รอบรับเข้า

ของชนิดเดียวกันอาจมีหลายรอบรับเข้า หลายวันหมดอายุ และอยู่คนละ Container ได้

ตัวอย่าง:

```text
ไข่ไก่
├── Lot A
│   ├── รับเข้า 1 ก.ค.
│   ├── 30 ฟอง
│   ├── หมดอายุ 12 ก.ค.
│   └── ตู้เย็น > ชั้น 2
│
└── Lot B
    ├── รับเข้า 5 ก.ค.
    ├── 60 ฟอง
    ├── หมดอายุ 20 ก.ค.
    └── ตู้เย็นสำรอง
```

ระบบต้องรู้ว่า:

* Item อะไร
* Lot ไหน
* รับเข้าเมื่อไร
* หมดอายุเมื่อไร
* เหลือเท่าไร
* อยู่ที่ Container ไหน
* ใครรับเข้า
* คงเหลือหลังรับเข้าเท่าไร

## 16. Reservation Workflow

ของที่สามารถยืมได้ต้องรองรับการจองล่วงหน้า

ตัวอย่าง:

```text
A ใช้รถอยู่ 08:00 - 12:00
B จองต่อ 12:00 - 15:00
```

ต้องรองรับ:

* ขอจอง
* อนุมัติ
* ปฏิเสธ
* ยกเลิก
* รอคิว
* พร้อมรับ
* หมดเวลาจอง

### กรณีคนก่อนยังไม่คืน

ถ้า A ยังไม่คืนเมื่อถึงเวลาของ B:

```text
B Status = Waiting For Return
```

ระบบควรแจ้ง:

* A ว่าเลยกำหนดและมีคนรอ
* B ว่าของยังไม่คืน
* ผู้ดูแลว่ามีคิวได้รับผลกระทบ

B สามารถ:

* รอต่อ
* เลื่อนเวลา
* ยกเลิก
* เลือกของทดแทน
* ติดต่อผู้ดูแล

## 27. UX Principle

ระบบต้องยืดหยุ่น แต่ไม่ซับซ้อน

ไม่ควรแสดงทุก Input พร้อมกัน

ใช้แนวคิด Progressive Disclosure:

```text
ผู้ใช้เลือกสิ่งที่ต้องการ
→ ระบบแสดงเฉพาะ Field ที่เกี่ยวข้อง
```

อาจใช้ Template เช่น:

* ของรายชิ้น
* ของให้ยืม
* วัสดุสิ้นเปลือง
* ของสด / หมดอายุ
* อุปกรณ์มีประกัน
* รถ / เครื่องจักร
* กำหนดเอง

Template เป็นตัวช่วยตั้งค่า ไม่ใช่การล็อกระบบแบบแข็ง

## 28. Core Business Rules

* รองรับหลาย Workspace
* Workspace แยกข้อมูลกัน
* รองรับการเชิญสมาชิก
* รองรับ Role หลัก
* รองรับ Permission พิเศษ
* คนเดียวทำหลายขั้นตอนได้
* อนุมัติรายการตัวเองได้ถ้ามี Permission
* ไม่มี Self-Service Flow แยก
* ใช้ Workflow เดียวกันทุกองค์กร
* Container ซ้อนกันได้หลายระดับ
* จำกัดการเห็น Container รายสมาชิกได้
* Search ต้องเคารพ Container Access
* รองรับของรายชิ้น
* รองรับของตามจำนวน
* รองรับหลายหน่วย
* Conversion ผูกกับ Item
* รองรับ Batch / Lot
* รองรับวันหมดอายุ
* รองรับวันหมดประกัน
* รองรับรอบบำรุงรักษา
* รองรับกำหนดเอง
* รองรับยืม
* รองรับคืน
* รองรับเบิก
* รองรับจอง
* รองรับแจ้งหาย
* รองรับแจ้งซ่อม
* รองรับจำหน่าย
* รองรับตรวจนับ
* รองรับปรับยอด
* ทุก Action สำคัญต้องมีประวัติ
* มี Notification
* มี Report

## 29. One Sentence Summary

WhereIs คือระบบบริหารสิ่งของแบบ Workspace ที่ช่วยให้รู้ว่ามีของอะไร อยู่ที่ไหน มีเท่าไร อยู่กับใคร ใครกำลังใช้ ใครรอใช้ต่อ ต้องคืนเมื่อไร ใกล้หมดอายุหรือถึงกำหนดอะไร พร้อม Workflow สำหรับยืม คืน เบิก จอง รับเข้า เติมสต็อก ตรวจนับ ปรับยอด แจ้งหาย ซ่อม จำหน่าย และควบคุมทุกขั้นตอนด้วย Role, Permission และสิทธิ์การเข้าถึง Container
